#ActorActionEndpoint / detail
Phase A — Authentication (Keycloak handles it; Party just stores the truth)
1 FSR UI → Keycloak User submits email + password; UI requests a token. POST /realms/tenant-btcl-btcl/protocol/openid-connect/token
2 Keycloak → Party SPI Keycloak delegates user lookup + credential validation to the party-user-storage provider (jar deployed in Keycloak). The provider calls Party over HTTP with a shared-secret header. GET /api/v1/internal/kc/users/by-username
POST /api/v1/internal/kc/users/validate-credentials
3 Keycloak → FSR UI Token issued; Keycloak protocol mappers inject operatorId, tenantId, partnerId, roles[], permissions[] as claims. { access_token, refresh_token, id_token }
Phase B — Write path (admin creates / updates / deletes a partner, user, role, etc.)
4 FSR UI → Party REST UI calls Party with the Keycloak bearer token. Party verifies the token against Keycloak JWKS and checks the caller's scope/tenantId claim. POST /api/v1/tenants/5/partners
Authorization: Bearer <kc JWT>
5 Party → party_master Single transaction: INSERT the partner row + INSERT a tenant_sync_job row (status=PENDING). Both committed before response. INSERT INTO partner (...)
INSERT INTO tenant_sync_job (entityType=PARTNER, op=CREATE, ...)
6 Party → Temporal After the transaction commits, Party starts a SyncEntityWorkflow on the party-sync-queue. Non-blocking; UI gets 200 immediately. WorkflowClient.start(wf::sync, 5, PARTNER, "42", CREATE, jobId)
Phase C — Async projection to the per-tenant DB (Temporal drives this; can be seconds later)
7 Temporal → Party worker Any one of the 3 active-active Party instances pulls the task (all 3 register on all 3 queues). Worker invokes TenantWriteActivity. activity.apply(5, PARTNER, CREATE, "42")
8 Worker → party_master Activity reads the latest master row (handles the case where CREATE then UPDATE arrive out of order: always projects current master state). SELECT * FROM partner WHERE id = 42
9 Worker → tenant DB Idempotent upsert against btcl_1_btcl_1.partner so retries are safe. On success, SyncAuditActivity.markSuccess(jobId) flips the job row to SUCCESS. INSERT INTO partner (...) ON DUPLICATE KEY UPDATE ...
Phase D — Read path (FSR UI reads via FSR backend; only Keycloak ever calls Party)
10 FSR backend → tenant DB FreeSwitch REST Spring Boot service queries its local tenant DB for partners / users / roles / permissions. No network hop to Party. Eventual consistency window = Phase C duration. SELECT * FROM partner WHERE partner_name = ?
11 FSR UI ← FSR backend UI renders the data that just landed. If Temporal lag is visible, UI shows a small ⋯ syncing chip next to the row; once Phase C completes the chip becomes ✓ synced. GET /fsrest/partners (FSR-owned endpoint)
Invariants to remember:
  • Party is the source of truth; the tenant DB is a projection. Reads from the tenant DB may lag by a few seconds after a write.
  • Only Keycloak calls /internal/kc/* — these endpoints are guarded by a shared secret and must not be exposed to the public LB.
  • Only admins call Party's CRUD via the UI (FSR UI or the dedicated Party admin UI). FSR runtime services never call Party.
  • Same sequence for auth_user, auth_role, auth_permission, auth_user_role, auth_role_permission, ip_access_rule, ui_menu_permission, partner_extra — only the entity type in step 5/8/9 changes; the flow is identical.
  • DELETE goes the same route: master row marked / deleted → sync job → projection DELETE. Retry semantics identical because all projection SQL is idempotent.
